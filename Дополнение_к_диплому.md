# Дополнение к пояснительной записке дипломного проекта

## 2.6. Тестирование информационной системы

### а) Разработка тестовых сценариев

Для проверки функциональности разработанной информационной системы «Beat Shop» был подготовлен набор тестовых сценариев, охватывающий ключевые пользовательские и административные процессы.

Основные цели тестирования:
- подтверждение корректности бизнес-логики;
- проверка целостности данных при выполнении операций CRUD;
- подтверждение корректной авторизации и разграничения прав доступа;
- проверка стабильности пользовательских сценариев от регистрации до покупки трека.

#### Набор базовых тестовых сценариев

1. **Регистрация нового пользователя**  
   **Предусловия:** пользователь не зарегистрирован в системе.  
   **Шаги:** открыть страницу регистрации → заполнить валидные поля (e-mail, пароль) → отправить форму.  
   **Ожидаемый результат:** создана новая запись пользователя, выполняется вход в систему или отображается сообщение об успешной регистрации.

2. **Авторизация пользователя**  
   **Предусловия:** учетная запись существует.  
   **Шаги:** перейти на страницу входа → ввести корректные учетные данные → нажать «Войти».  
   **Ожидаемый результат:** пользователь получает доступ к личному кабинету, в интерфейсе отображается его имя/профиль.

3. **Просмотр каталога треков и фильтрация**  
   **Предусловия:** в базе данных есть опубликованные треки.  
   **Шаги:** открыть главную страницу → применить фильтр по жанру/автору → выполнить сортировку по популярности.  
   **Ожидаемый результат:** список треков обновляется в соответствии с выбранными параметрами.

4. **Добавление трека в избранное**  
   **Предусловия:** пользователь авторизован, в каталоге есть трек.  
   **Шаги:** открыть карточку трека → нажать кнопку «В избранное».  
   **Ожидаемый результат:** трек появляется в разделе «Избранное», в БД создается запись связи user-track.

5. **Покупка трека**  
   **Предусловия:** пользователь авторизован, трек доступен для покупки.  
   **Шаги:** открыть страницу трека → нажать «Купить» → подтвердить действие.  
   **Ожидаемый результат:** формируется заказ/покупка, пользователю становится доступно скачивание, в БД фиксируется транзакция.

6. **Проверка доступа в административную панель**  
   **Предусловия:** есть пользователь с ролью admin и обычный пользователь.  
   **Шаги:** выполнить вход под обычной учетной записью и открыть URL админ-панели; затем повторить под admin.  
   **Ожидаемый результат:** для обычного пользователя доступ запрещен (403/редирект), администратор получает доступ к функциям управления контентом.

7. **Создание и редактирование карточки трека администратором**  
   **Предусловия:** выполнен вход как admin.  
   **Шаги:** открыть форму создания → заполнить название, жанр, BPM, цену → сохранить → открыть редактирование и изменить цену.  
   **Ожидаемый результат:** карточка успешно создается и обновляется, изменения корректно отображаются в каталоге.

#### Критерии завершения тестирования

Тестирование считается успешно завершенным при выполнении условий:
- все критические сценарии (регистрация, вход, покупка, доступ к контенту) имеют статус **Passed**;
- не обнаружены блокирующие дефекты (Severity: Critical);
- остаточные дефекты не влияют на основной пользовательский поток.

### б) Тестирование информационной системы при помощи библиотеки Selenium

Для автоматизации проверки пользовательских сценариев использована библиотека **Selenium WebDriver**. Автотесты позволяют воспроизводимо запускать проверку интерфейса после изменений в коде и снижать риск регрессий.

#### Цели автоматизированного UI-тестирования
- проверка работоспособности основных экранов в браузере;
- проверка корректности взаимодействия фронтенда и backend API;
- быстрое выявление ошибок после обновления функционала.

#### Пример структуры автотестов

Рекомендуемая структура набора Selenium-тестов:
- `test_auth.py` — проверка входа/выхода;
- `test_catalog.py` — каталог треков, фильтры и поиск;
- `test_favorites.py` — добавление и удаление избранного;
- `test_purchase.py` — сценарий покупки трека;
- `test_admin.py` — базовые операции админ-панели.

#### Пример тест-кейса на Selenium (авторизация)

```python
from selenium import webdriver
from selenium.webdriver.common.by import By
from selenium.webdriver.support.ui import WebDriverWait
from selenium.webdriver.support import expected_conditions as EC


def test_login_success():
    driver = webdriver.Chrome()
    wait = WebDriverWait(driver, 10)

    try:
        driver.get("http://localhost:5173/login")

        email = wait.until(EC.visibility_of_element_located((By.NAME, "email")))
        password = driver.find_element(By.NAME, "password")
        submit = driver.find_element(By.CSS_SELECTOR, "button[type='submit']")

        email.send_keys("user@example.com")
        password.send_keys("StrongPassword123")
        submit.click()

        wait.until(EC.url_contains("/profile"))
        assert "/profile" in driver.current_url
    finally:
        driver.quit()
```

#### Организация запуска

1. Запустить backend и frontend приложения в тестовом окружении.  
2. Подготовить тестовые данные (пользователь, жанры, треки).  
3. Выполнить запуск:  
   `pytest tests/ui -v`  
4. Сформировать отчет о прохождении (например, в формате Allure или JUnit XML).

#### Результаты и выводы по Selenium-тестированию

Автоматизированное тестирование подтвердило корректность выполнения основных пользовательских операций. Selenium-тесты рекомендуется включить в процесс CI/CD для автоматической проверки стабильности системы после каждого изменения.

## 2.7. Руководство пользователя

### Назначение системы

Информационная система «Beat Shop» предназначена для поиска, прослушивания и покупки музыкальных треков. Пользователь взаимодействует с веб-интерфейсом и получает доступ к персонализированным функциям: избранное, история покупок, персональный профиль.

### Минимальные требования
- современный браузер (Google Chrome, Mozilla Firefox, Microsoft Edge);
- доступ к сети Интернет;
- включенный JavaScript.

### Инструкция для пользователя

#### 1. Регистрация и вход
1. Откройте главную страницу сервиса.
2. Нажмите кнопку **«Регистрация»**.
3. Заполните поля e-mail и пароль.
4. После успешной регистрации выполните вход через форму **«Войти»**.

#### 2. Поиск и просмотр треков
1. Перейдите в раздел каталога.
2. Используйте строку поиска по названию или автору.
3. Примените фильтры (жанр, BPM, цена) для уточнения выдачи.
4. Откройте карточку трека для просмотра подробной информации.

#### 3. Добавление в избранное
1. Авторизуйтесь в системе.
2. Нажмите иконку/кнопку **«В избранное»** в карточке трека.
3. Проверьте сохраненный список в личном кабинете.

#### 4. Покупка трека
1. Откройте нужный трек.
2. Нажмите **«Купить»**.
3. Подтвердите действие.
4. После успешной покупки перейдите в раздел **«Мои покупки»** и скачайте файл.

#### 5. Работа с профилем
1. Откройте раздел **«Профиль»**.
2. При необходимости обновите данные аккаунта.
3. Просмотрите историю операций (покупки/прослушивания, если предусмотрено интерфейсом).

### Возможные проблемы и решения

- **Не удается войти в систему:** проверьте правильность e-mail и пароля, раскладку клавиатуры, отключите автозаполнение и введите данные вручную.
- **Трек не добавляется в избранное:** убедитесь, что выполнен вход в учетную запись.
- **Не отображается список треков:** обновите страницу, проверьте подключение к интернету и доступность backend-сервиса.
- **Не скачивается купленный файл:** проверьте, завершена ли операция покупки, и повторите скачивание из раздела «Мои покупки».

### Рекомендации по безопасной работе
- использовать сложные пароли и не передавать их третьим лицам;
- выполнять выход из системы на общедоступных устройствах;
- регулярно обновлять браузер до актуальной версии.

## 4. Разработка информационной системы (дополнительный материал)

### 4.1. Архитектура решения

Разработанная система построена по клиент-серверной архитектуре и состоит из двух ключевых подсистем:
- **Frontend (клиентское приложение):** обеспечивает пользовательский интерфейс, навигацию, отображение каталога и взаимодействие с API.
- **Backend (серверное приложение):** реализует бизнес-логику, авторизацию, работу с данными, операции покупки, избранного и администрирования.

Взаимодействие между подсистемами выполняется через HTTP API. Такой подход упрощает масштабирование, поддержку и развитие проекта.

### 4.2. Реализация серверной части

Серверная часть реализована с использованием современного Python-стека. В проекте выделены слои:
- **API-слой** (маршруты) — принимает запросы, валидирует входные данные и возвращает ответы;
- **слой бизнес-логики/репозиториев** — инкапсулирует операции над сущностями предметной области;
- **слой доступа к данным** — взаимодействует с реляционной базой данных.

Основные реализованные сущности:
- пользователь;
- автор;
- трек;
- жанр;
- покупка;
- избранное;
- подписка (при наличии в конфигурации).

Для эволюции схемы базы данных применяются миграции, что обеспечивает управляемое изменение структуры таблиц и воспроизводимость развёртывания.

### 4.3. Реализация клиентской части

Клиентская часть реализована как одностраничное веб-приложение (SPA). Интерфейс обеспечивает:
- каталог музыкальных треков;
- карточку трека с ключевыми параметрами;
- формы регистрации и авторизации;
- личный кабинет пользователя;
- административные формы управления данными.

Использование компонентного подхода позволило повысить переиспользуемость UI-элементов и упростить сопровождение кода.

### 4.4. Обеспечение безопасности

В системе реализованы базовые меры информационной безопасности:
- аутентификация и контроль доступа к защищенным разделам;
- разграничение ролей (пользователь/администратор);
- проверка входных данных и обработка ошибочных запросов;
- ограничение доступа к административным маршрутам.

При промышленной эксплуатации рекомендуется дополнительно внедрить:
- rate limiting на критичных эндпоинтах;
- централизованный аудит операций;
- регулярное сканирование зависимостей на уязвимости.

### 4.5. Интеграция и развёртывание

Для развертывания системы целесообразно использовать разделение окружений:
- **development** — активная разработка и отладка;
- **staging** — предпродакшн-проверка;
- **production** — эксплуатационный контур.

Минимальный pipeline поставки включает этапы:
1. установка зависимостей;
2. запуск тестов;
3. применение миграций;
4. публикация backend и frontend;
5. проверка доступности сервиса (smoke-check).

### 4.6. Перспективы развития системы

В качестве направлений дальнейшего развития можно выделить:
- внедрение рекомендательной подсистемы на основе истории прослушиваний и покупок;
- расширение вариантов оплаты и формирование закрывающих документов;
- интеграцию аналитической панели для авторов;
- добавление многоязычного интерфейса;
- реализацию мобильной версии или нативного приложения.

Предложенные улучшения повышают практическую ценность системы и расширяют ее применимость для реальных сценариев цифровой дистрибуции музыкального контента.
